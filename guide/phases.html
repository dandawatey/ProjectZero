Phase Lifecycle

Every product progresses through a governed, sequential pipeline. Each phase is backed by a Temporal workflow that orchestrates agent work, produces artifacts, and gates on human approval before advancing.

PHASE 0

Factory Setup

One-time initialization of the ProjectZero platform. Stands up infrastructure (Postgres, Temporal), runs migrations, validates environment, and creates the .factory/ configuration directory.

Command /factory-init
Workflow FactoryInitWorkflow
What happens Docker containers start, DB schema created, config validated

To-Do Checklist

  • Install Docker and verify daemon is running Required to run Postgres, Temporal, and backend containers locally.
  • Copy .env.example → .env and fill all required vars Set DB URL, Temporal address, API keys, and JIRA/Confluence credentials.
  • Start containers: docker compose up -d Brings up Postgres, Temporal server, and Temporal worker in background.
  • Run database migrations: alembic upgrade head Creates all factory tables — workflows, brain, approvals, artifacts.
  • Validate Temporal worker connects Check worker logs for "Worker started" — if absent, check TEMPORAL_ADDRESS env var.
  • Run /factory-init and confirm green status Runs health checks on all integrations and seeds factory configuration.
PHASE 1

Product Creation

Creates a new product workspace. Registers the product in Postgres, scaffolds the directory structure, and initializes the phase pipeline. The product is now ready for discovery.

Command /bootstrap-product
Workflow BootstrapProductWorkflow
What happens Product record created, workspace scaffolded, pipeline initialized

To-Do Checklist

  • Decide product name and slug Slug must be lowercase, hyphen-separated, unique across the factory (e.g. my-product).
  • Run /bootstrap-product <name> Scaffolds directory structure, creates DB record, and initializes the phase pipeline.
  • Confirm product record in DB Call GET /api/v1/products — new product should appear with status created.
  • Verify products/<name>/ directory scaffolded Should contain .claude/, src/, tests/, and CLAUDE.md stubs.
  • Set product-specific config and JIRA project key Update .env or product config with JIRA key so tickets are created in the right project.
  • Confirm Brain memory initialized Factory writes a seed memory entry for the product — visible at GET /api/v1/brain/memories.
  • Initialize git repository (if not already present) Run git init && git commit --allow-empty -m "chore: init product repo" — required before Vision-to-PRD starts committing artifacts.
  • Confirm design system initialized (Step 10 of 13) Auto-runs as part of /bootstrap-product. Verify src/design-system/tokens.ts, motion.ts, and 6 base components exist. Run npm run storybook to confirm Storybook starts at localhost:6006. Use --skip=design-system flag to bypass if not needed.
PHASE 2a

Vision → PRD

Transforms a high-level product vision into a structured PRD. The agent conducts a discovery interview, researches the market, analyzes competitors, identifies target users, and drafts a comprehensive product requirements document.

Command /vision-to-prd
Workflow VisionToPRDWorkflow
What happens Discovery interview, market research, competitor analysis, PRD generation

To-Do Checklist

  • Write a 1–3 sentence product vision statement Captures the core problem, target user, and desired outcome — input to the discovery interview.
  • Run /vision-to-prd and complete discovery interview Agent asks structured questions about users, goals, and constraints to shape the PRD.
  • Review generated PRD — problem statement, goals, non-goals Ensure the problem is specific, goals are measurable, and non-goals explicitly bound scope.
  • Validate target user personas and pain points Check that personas reflect real users, not assumptions — adjust if needed before architecture starts.
  • Review competitor analysis and market sizing Confirm key competitors are identified and differentiation is clear.
  • Iterate with follow-up prompts until PRD is accurate Use chat mode to refine sections — the PRD is the source of truth for all downstream phases.
  • Run /approve to advance to Phase 2b / Phase 3 Locks the PRD and signals the workflow to proceed.
PHASE 2b

Business Discovery

Deep business analysis that runs alongside or after Vision-to-PRD. Produces business model canvas, revenue projections, go-to-market strategy, competitive moat analysis, and risk assessment documents.

Command /business-docs
Workflow BusinessDiscoveryWorkflow
What happens Business model, revenue model, GTM strategy, risk analysis generated

To-Do Checklist

  • Run /business-docs with PRD as input Agent reads the approved PRD and generates the full business analysis suite.
  • Review business model canvas Validate value proposition, customer segments, channels, and revenue streams are realistic.
  • Validate revenue projections and assumptions Check that TAM/SAM/SOM figures and conversion assumptions are defensible.
  • Review go-to-market strategy and launch sequencing Confirm target channels, launch phases, and success metrics are defined.
  • Review competitive moat analysis Ensure differentiation is specific — not generic claims like "better UX".
  • Identify and accept top 3 risks with mitigations Each risk needs an owner, a likelihood score, and a concrete mitigation plan.
  • Approve or request revision of business docs Business docs are an input to Phase 7 — approve now to allow revision after launch.
PHASE 3

Specification

Decomposes the approved PRD into a detailed technical specification. Generates user stories with acceptance criteria, defines API contracts, data models, and creates a dependency graph for implementation ordering.

Command /spec
Workflow SpecificationWorkflow
What happens Stories created, API contracts defined, data models specified, dependency graph built

To-Do Checklist

  • Confirm PRD is approved (Phase 2a exit gate passed) Spec cannot start without an approved PRD — the workflow enforces this gate.
  • Run /spec Agent decomposes the PRD into stories, API contracts, data models, and a dependency graph.
  • Review all generated user stories Every story must have clear acceptance criteria in Given/When/Then format.
  • Validate API contracts Check endpoint names, HTTP methods, request/response schemas, and error codes match the PRD intent.
  • Review data models and field constraints Confirm required fields, nullability, foreign keys, and index candidates are specified.
  • Check dependency graph — ordering makes sense Foundational stories (auth, DB schema) must precede dependent feature stories.
  • Create / sync JIRA tickets for all stories Use /ticket-create or the JIRA integration to push all stories into the active sprint.
  • Create mandatory OBS-001..010 observability tickets Before any feature work: OBS-001 structured logging, OBS-002 audit middleware, OBS-003 /health endpoint, OBS-004 distributed tracing, OBS-005 metrics, OBS-006 error tracking, OBS-007 DB query logging, OBS-008 background job observability, OBS-009 auth event audit, OBS-010 performance baseline.
  • Run /approve to advance to Phase 4 Locks the spec — changes after approval require a change request ticket.
PHASE 4

Architecture

Designs the system architecture based on the specification. Selects technology stack, defines service boundaries, designs database schema, plans infrastructure, and produces architecture decision records (ADRs).

Command /arch
Workflow ArchitectureWorkflow
What happens Tech stack selected, services designed, DB schema planned, ADRs written

To-Do Checklist

  • Confirm specification is approved (Phase 3 exit gate passed) Architecture depends on a locked spec — starting before approval risks re-work.
  • Run /arch Agent generates tech stack recommendation, service design, DB schema, and ADRs.
  • Review technology stack selection and rationale Ensure chosen stack aligns with team expertise and long-term maintenance costs.
  • Validate service boundaries and component diagram Check that components are cohesive, dependencies flow one way, and no circular references exist.
  • Review database schema — indexes, constraints, relations Confirm foreign keys, unique constraints, and indexes for high-query columns are present.
  • Read all ADRs — decisions and rejected alternatives documented Each ADR must record what was chosen, what was rejected, and why — no undocumented choices.
  • Review infrastructure plan (compute, storage, networking) Validate sizing estimates, redundancy requirements, and environment parity (dev/staging/prod).
  • Define observability architecture as ADRs Record decisions for structured logging strategy (correlation IDs), metrics collection (Prometheus), distributed tracing (OpenTelemetry), error tracking, and SLO targets. These govern OBS-001..010 implementation tickets in Phase 5.
  • Run /approve to advance to Phase 5 Locks architecture — structural changes after approval require a new ADR and re-approval.
PHASE 5

Implementation

Executes the implementation plan story by story. Agents generate code, write tests, create components, and wire up integrations. Each story goes through its own create-validate-approve cycle before the next begins.

Command /implement
Workflow ImplementationWorkflow
What happens Code generation, test writing, component creation, story-by-story delivery

To-Do Checklist

  • Confirm architecture is approved (Phase 4 exit gate passed) Implementation cannot begin without an approved architecture — no freelance coding.
  • Run /implement — agent picks highest-priority story Agent reads dependency graph and backlog order to select the correct story automatically.
  • Confirm tests are written BEFORE implementation (TDD) Failing tests must exist and be committed before any implementation code is written.
  • Run /check after each story Enforces ≥ 80% test coverage, zero lint errors, and zero type errors before story is marked done.
  • Review code diff for each story before approving Apply the 4-eye principle — maker implements, checker/reviewer approve before next story starts.
  • Update JIRA ticket status as stories complete Use /ticket or the JIRA integration to move tickets to In Review → Done.
  • Repeat until all stories implemented and approved Every story in the spec must be delivered — partial delivery is not an exit condition.
  • Run /approve to advance to Phase 6 Signals the ImplementationWorkflow to complete and triggers QualityReleaseWorkflow.
PHASE 6

Quality + Release

Comprehensive quality assurance and release preparation. Runs integration tests, performs security audits, checks accessibility compliance, optimizes performance, and prepares release artifacts including changelogs and deployment configs.

Command /release
Workflow QualityReleaseWorkflow
What happens Integration tests, security audit, a11y check, performance optimization, release prep

To-Do Checklist

  • Confirm all implementation stories approved (Phase 5 exit gate) Release cannot proceed if any story is still pending — check JIRA for open tickets.
  • Run /release Triggers QualityReleaseWorkflow — runs integration tests, security audit, a11y check, and perf scan.
  • Review integration test results — all green All API endpoint tests, DB migration tests, and end-to-end flows must pass.
  • Review security audit findings No critical or high-severity findings may remain unresolved — medium findings need an owner.
  • Verify accessibility compliance — Lighthouse ≥ 90 Run lighthouse --only-categories=accessibility and fix any WCAG AA violations.
  • Review performance report — Core Web Vitals within targets LCP < 2.5s, CLS < 0.1, FID < 100ms on production-equivalent environment.
  • Review generated changelog and release notes Changelog must reference all ticket IDs and summarize user-facing changes accurately.
  • Tag release version and push to registry Follow semver — use git tag v1.0.0 and push Docker image to container registry.
  • Run /approve to advance to Phase 7 / deploy Final gate before production deployment — requires explicit human sign-off.
PHASE 7

Business Planning

Post-build business documentation. Generates launch plan, pricing strategy, marketing copy, investor pitch materials, and operational runbooks. Synthesizes all business docs into a cohesive go-to-market package.

Command /business-docs
Workflow BusinessPlanningWorkflow
What happens Launch plan, pricing, marketing copy, pitch deck, runbooks generated

To-Do Checklist

  • Run /business-docs with completed product as context Agent reads the shipped product state and generates updated, launch-ready business docs.
  • Review launch plan — timeline, channels, success metrics Confirm launch date, owned channels (email, social, press), and how success is measured at 30/90 days.
  • Validate pricing tiers and packaging decisions Check that free/paid tiers, usage limits, and upgrade paths are clearly defined and implemented.
  • Review marketing copy — headlines, value props, CTAs Landing page copy must match the actual product capabilities — no vaporware claims.
  • Review investor pitch deck structure and narrative Problem → solution → traction → market → team → ask — each slide must be evidenced.
  • Review operational runbooks — on-call, incident, escalation On-call engineers need step-by-step playbooks for the top 5 most likely failure modes.
  • Publish approved docs to Confluence Run /publish-cxo to push business docs to the CxO dashboard space in Confluence.
PHASE 8

Operations

Ongoing operational monitoring and maintenance. Sets up health checks, alerting, log aggregation, and automated incident response. The product enters a steady-state where agents monitor and respond to issues.

Command /monitor
Workflow OperationsWorkflow
What happens Health checks configured, alerts set, monitoring dashboards created

To-Do Checklist

  • Run /monitor to activate ops agents Starts the OperationsWorkflow — ops agents begin polling health endpoints and log streams.
  • Confirm health check endpoints respond Call GET /health and GET /api/v1/confluence/health — all integrations must return ok.
  • Set alerting thresholds Define error rate (> 1%), p99 latency (> 2s), and uptime (< 99.5%) thresholds for PagerDuty/Slack alerts.
  • Verify log aggregation pipeline is ingesting events Confirm application logs flow into the aggregator (Loki / CloudWatch / Datadog) within 30s.
  • Review monitoring dashboards — all key metrics visible Confirm request rate, error rate, latency, DB query time, and queue depth panels are populated.
  • Test incident response runbook with a dry-run scenario Simulate a degraded DB connection — confirm alerts fire, runbook steps are clear, and escalation works.
  • Confirm on-call rotation and escalation contacts configured Primary and secondary on-call must be set with phone + Slack — no single point of failure.
  • Schedule first post-launch retrospective Book within 2 weeks of go-live — captures learnings while context is fresh and feeds the Brain.

Phase Transitions

Every phase transition follows the same governance pattern:

  1. Workflow completes
    All activities in the phase finish and artifacts are produced.
  2. Artifacts persisted
    sync_state() writes results to Postgres.
  3. Approval gate
    Workflow pauses on a Temporal signal-wait.
  4. Human review
    User reviews artifacts in Control Tower or CLI.
  5. Approval signal
    /approve sends the signal, workflow resumes.
  6. Next phase starts
    The subsequent workflow is automatically triggered.
Recovery

If a phase needs to be re-run after feedback, use /recover-workflow to reset and restart it. Previous artifacts are preserved in version history.
