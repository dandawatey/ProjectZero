How Development Happens

Agent-Driven Development Workflow

Agents autonomously build features through TDD and MCRA gates. Every feature traces to JIRA, passes automated quality checks, gets reviewed, and is approved before merge.

Development Overview

When you bootstrap a new product (e.g., i-comply) with /bootstrap-product --enterprise-features tenant-isolation,multi-db,encryption, you get:

  • JIRA Project with 13 enterprise tickets (P0, P1, P2, P3)
  • Agent Team (11 agents ready to work)
  • MCRA Workflow (Maker → Checker → Reviewer → Approver gates)
  • Git Repository configured for feature branches + PRs

Then, for each JIRA ticket, you invoke agents:

/agent-work \ --ticket ICOMPLY-1 \ --product i-comply \ --repo https://github.com/org/i-comply.git \ --user yogesh@example.com

Agents autonomously:

  • Clone repo, read acceptance criteria
  • TDD cycle: write tests → implement → refactor
  • Commit & push with ticket reference
  • Open PR on GitHub
  • Pass quality gates (Checker: tests, coverage, lint, types)
  • Get code reviewed (Reviewer: quality, architecture, tests)
  • Merge to main (Approver: business sign-off)

MCRA Gates (4-Step Workflow)

🔵 MAKER
Backend/Frontend/Data Engineer
Writes code (TDD)
→
🟢 CHECKER
Automated quality gates
Tests, coverage, lint, types
→
🟡 REVIEWER
Code review agent
Quality, architecture, tests
→
🟠 APPROVER
Leadership (CTO/CEO)
Final sign-off + merge

Gate 1: MAKER (Backend Engineer + Specialists)

Responsibility: Write code that passes all acceptance criteria.

Process:

  1. Clone repo from GitHub
  2. Create feature branch (e.g., icomply-1-tenant-isolation)
  3. Write tests first (TDD RED phase) — tests should FAIL
  4. Implement code (TDD GREEN phase) — tests should PASS
  5. Refactor (TDD REFACTOR phase) — improve code quality
  6. Commit with ticket reference: feat: tenant isolation (ICOMPLY-1)
  7. Push to GitHub
  8. Open PR with description + acceptance criteria
  9. Route to Checker

Example Output:

Branch: icomply-1-tenant-isolation Files: + tests/test_tenant_isolation.py (42 tests) + src/tenancy.py (RLS enforcement) + src/middleware/tenant_context.py (JWT → context) PR: https://github.com/org/i-comply/pull/1 Coverage: 85%

Gate 2: CHECKER (Automated Quality)

Responsibility: Verify code quality, coverage, and security.

Automated Checks:

  • ✅ Tests pass (pytest) — all tests must pass
  • ✅ Coverage ≥80% — minimum threshold enforced
  • ✅ Linting (ruff) — zero errors
  • ✅ Type checking (pyright) — zero errors
  • ✅ Security scan (OWASP ZAP) — no critical issues
  • ✅ Ticket reference — commit message includes ICOMPLY-1

If all pass → ✅ Proceed to Reviewer
If any fail → ❌ Loop back to Maker with specific error messages

Example Output:

✓ pytest: 42 passed in 5.2s ✓ coverage: 85% (≥80%) ✓ ruff: 0 errors ✓ pyright: 0 errors ✓ OWASP ZAP: 0 critical issues ✓ ticket ref: ICOMPLY-1 found Status: ✅ ALL GATES PASSED → Proceed to Reviewer

Gate 3: REVIEWER (Code Review Agent)

Responsibility: Review code quality, architecture, test coverage.

Review Checks:

  • ✅ Code quality — follows style guide, readable, maintainable
  • ✅ Architecture — matches ADR (Architecture Decision Records)
  • ✅ Test coverage — tests comprehensive, edge cases handled
  • ✅ Performance — no obvious performance issues

If approved → ✅ Proceed to Approver
If changes requested → 🔄 Maker updates code, re-review

Example Review Comment:

Code quality: ✓ Looks good. Well-structured RLS policies. Architecture: ✓ Matches tenancy ADR-001. Correct use of contextvars. Tests: ✓ 85% coverage. All edge cases tested: cross-tenant access blocked, audit logs created. Status: ✅ APPROVED → Proceed to Approver

Gate 4: APPROVER (Leadership)

Responsibility: Final business sign-off, governance check, merge.

Approval Checks:

  • ✅ Business requirements — acceptance criteria met
  • ✅ Governance — no security/compliance violations
  • ✅ Risk assessment — no breaking changes

If approved → ✅ Merge PR to main → Close JIRA ticket → Feature LIVE
If rejected → ❌ Explain reasons, ticket stays open

Example Approval:

Business: ✓ Tenant isolation required for compliance. Acceptance criteria met. Governance: ✓ RLS enforced. No security issues. Audit logging enabled. Risk: ✓ No breaking changes. Backward compatible. Status: ✅ APPROVED Action: Merging PR to main... Action: Closing ICOMPLY-1 (status: Done) ✅ Feature LIVE in production

Key Differences from Manual Development

❌ Manual: "I'll write code now"

Human writes code, then writes tests, then commits. No discipline on TDD.

✅ Agent: "Test first, then code"

Agent writes tests first (RED), then code (GREEN), then refactors. Always TDD.

❌ Manual: "Merge when I think it's done"

Code merged without automated quality checks. Coverage can be 10-50%.

✅ Agent: "Only merge if tests + coverage + lint + types pass"

Automated gates enforce ≥80% coverage, zero lint/type errors. No exceptions.

❌ Manual: "I'll optimize later"

Code shipped with tech debt, performance issues, no docs.

✅ Agent: "Refactor while TDD, document in code, optimize per ADR"

Refactoring built into TDD cycle. Architecture documented. Decisions traced.

❌ Manual: "I'll push to main, figure it out"

Direct commits to main. Breaks production. Ticket status unclear.

✅ Agent: "Feature branch → PR → 4 gates → merged only when all pass"

Every commit traced to JIRA. PR visible. All gates enforced. Clear audit trail.
