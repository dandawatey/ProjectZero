System Architecture

ProjectZero is a layered platform where a React dashboard observes activity orchestrated by Temporal workflows, mediated through a FastAPI backend, and persisted in Postgres.

Components

React Control Tower

Real-time dashboard for monitoring workflows, agent activity, approval queues, and product status. Connects to the backend via REST and WebSocket for live updates.

FastAPI Backend

Central API layer. Handles product CRUD, phase transitions, approval gates, workflow triggers, and serves the Control Tower. Runs async with uvicorn.

PostgreSQL

Source of truth for all product state: phases, approvals, audit logs, business documents, stories, and configuration. Schema-managed via Alembic migrations.

Temporal Server

Durable workflow orchestration. Every phase is a Temporal workflow with retry policies, timeouts, and human-signal activities. Guarantees exactly-once execution.

Agent Workers

Temporal workers that execute activities. Each worker hosts specialized agent logic: research, spec writing, architecture design, code generation, quality checks.

Claude Code CLI

The developer interface. Slash commands trigger workflows, pass context, and relay approvals. Agents run as Claude Code sessions with tool access and file I/O.

Data Flow

Here is the typical path of a request through the system:

  • User runs a command — e.g., /vision-to-prd in Claude Code CLI
  • CLI calls FastAPI — POST to /api/products/{id}/workflows/vision-to-prd
  • FastAPI starts a Temporal workflow — VisionToPRDWorkflow is registered and dispatched
  • Temporal schedules activities — Market research, competitor analysis, PRD drafting are queued as activities
  • Agent workers execute — Claude-powered agents run each activity, producing artifacts
  • Artifacts written to Postgres — PRD, research data, and metadata are persisted via FastAPI
  • Workflow hits approval gate — Temporal sends a signal-wait, pausing until human approval
  • Control Tower shows pending approval — React dashboard polls and displays the gate
  • User approves — /approve sends a signal to Temporal, workflow resumes

State Synchronization Pattern

Key Pattern

Temporal is the orchestration authority. Postgres is the persistence authority. They stay in sync through a deliberate sync pattern.

The sync flow works as follows:

Temporal Workflow (executes phase)
    |
    |-- Activity: do_work()
    |       |-- Agent performs task
    |       |-- Returns result
    |
    |-- Activity: sync_state()
    |       |-- POST /api/internal/sync
    |       |-- FastAPI writes to Postgres
    |       |-- Phase status, artifacts, timestamps updated
    |
    |-- Activity: wait_for_approval()
    |       |-- Temporal signal-wait
    |       |-- FastAPI relays /approve as signal
    |
    |-- Activity: sync_state()  (after approval)
            |-- Final state persisted
            |-- Next phase unlocked

This pattern ensures that even if the FastAPI server restarts, Temporal retains the full execution state and can resume. And even if Temporal restarts, Postgres has the last-known state for the Control Tower to display.

Why Not Just Use Postgres?

Postgres alone cannot orchestrate long-running workflows with retries, timeouts, and human-in-the-loop signals. Temporal provides durable execution guarantees that a database and background job queue cannot match. The combination gives you both reliable orchestration (Temporal) and queryable state (Postgres).

Hook System

Claude Code hooks automate memory, graph refresh, and execution telemetry without manual intervention. Hooks are configured in .claude/settings.json and fire at defined lifecycle events.

SessionStart Hook

Fires when a new Claude Code session opens. Runs claude_mem_session_start.sh to ensure the claude-mem worker is live on port 37777 before any agent work begins.

PostToolUse (Write/Edit)

Fires after any file write or edit. Triggers graphify-ts generate to keep the knowledge graph current, then records the observation in claude-mem.

PostToolUse (Bash)

Fires after Bash tool use. Emits execution events via post_tool_use.sh and records the observation in claude-mem for future session recall.

Stop Hook

Fires when the session ends. Runs claude_mem_stop.sh to flush the compression buffer and promote high-value memories (score ≥0.7) to the Postgres Brain.

claude-mem in the System

claude-mem slots between the Claude Code CLI and the Postgres Brain as a fast local cache with compression.

Claude Code CLI
    |
    +-- PostToolUse hook ──> claude-mem worker (:37777)
    |                              |
    |                              |-- SQLite local store (CLAUDE_MEM_DB_PATH)
    |                              |-- /mem-search (3-layer query)
    |
