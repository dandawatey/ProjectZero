Governance Rules

Non-negotiable rules that keep every build honest, auditable, and production-safe.

Core Rules

🛑
No Workflow → No Build
Every piece of work must be tracked in a Temporal workflow. If there is no workflow ID, nothing gets built. No side-channel hacking.
🎫
No Ticket → No Work
Every workflow is linked to a ticket. No orphan work. If it does not have a ticket number, it does not exist.
🧪
TDD Mandatory
Tests are written before implementation. The test file must exist and fail before any production code is generated.
📊
80% Coverage Minimum
Code coverage must hit at least 80% across lines, branches, and functions. Below that, the quality gate blocks the workflow.
🔍
Everything Auditable
Every decision, approval, rejection, and retry is logged in the workflow history. Full traceability from ticket to deploy.
🔄
Everything Resumable
Any workflow can be paused, crashed, or interrupted and resumed from its last checkpoint. No work is ever lost.

Maker → Checker → Reviewer → Approver

Every deliverable passes through a four-step quality pipeline before it is considered done:

Maker
Agent creates the artifact
→
Checker
Automated validation
→
Reviewer
Peer agent review
→
Approver
Human sign-off

The Maker generates code or documentation. The Checker runs linting, tests, and coverage. The Reviewer (another agent) audits for correctness. The Approver (a human) gives final sign-off at stage gates.

No Fake Completion

A workflow is only "done" when every exit criterion is objectively met. Here is what counts and what does not:

"Done" means

  • All tests pass (zero failures)
  • Coverage at or above 80%
  • Linter reports zero errors
  • All stage gates approved
  • Artifacts committed to the repo
  • Workflow status is COMPLETED in Temporal

"Done" does NOT mean

  • Agent said "I'm done"
  • Code compiles but tests are skipped
  • Coverage is "close enough" at 78%
  • PR is open but not merged
  • Manual workaround applied
  • Workflow terminated manually

BMAD Methodology

BMAD (Business-Minded AI Development) is the guiding methodology. It ensures that every technical decision traces back to a business objective. Business docs (PRD, BRD, market analysis) are generated first. Only then do technical workflows begin. This prevents building the wrong thing well.

BMAD Principles

  • Business value drives every ticket
  • Stakeholder personas are defined before specifications
  • Acceptance criteria are measurable, not subjective
  • Every deliverable maps to a PRD requirement

SPARC Framework

SPARC defines the five stages every build workflow moves through:

S
Specification
→
P
Pseudocode
→
A
Architecture
→
R
Realization
→
C
Completion

Each stage produces artifacts that become inputs to the next. No stage can begin until the previous stage's gate is passed. This is enforced by Temporal workflow logic, not by convention.

Stage Gates

A stage gate is a checkpoint between SPARC stages. The workflow pauses and waits for approval before proceeding.

1
Spec Gate — Specification reviewed and approved. Requirements are locked.
2
Design Gate — Pseudocode and architecture signed off. Implementation plan is concrete.
3
Quality Gate — All tests pass, coverage met, linter clean. Code is review-ready.
4
Release Gate — Final human approval before the workflow is marked COMPLETED and artifacts are released.
