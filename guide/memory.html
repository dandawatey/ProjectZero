Memory

ProjectZero has two complementary memory layers: claude-mem for fast cross-session compression within a single machine, and the Brain (Postgres) for cross-product, cross-machine persistent memory. They sync automatically at session end.

Two-Layer Architecture

Developer machine
  Claude Code session
      |
      +-- claude-mem worker (:37777) ──── SQLite (fast, local, compressed)
      |         |
      |         | score >= 0.7 at Stop
      |         v
      +-- sync_to_brain.py ───────────── POST /api/v1/brain/memories
                                               |
                                        Postgres (shared, cross-product)

claude-mem (Local)

Runs as a local worker on port 37777. Stores observations in SQLite with compression. Fast, cheap, ~50 tokens per search. Scoped to the developer's machine.

Brain (Postgres)

Central REST API at /api/v1/brain/. Memories, decisions, patterns, and conversation history. Shared across all products and all agents. System of record.

claude-mem

claude-mem provides cross-session memory compression for Claude Code agents. Source: github.com/thedotmack/claude-mem.

Starting the Worker

# Manual start (auto-started by SessionStart hook)
npx claude-mem start
  Memory worker running on port 37777

# Check worker health
curl http://localhost:37777/health
  {"status":"ok","version":"x.y.z"}
Tip

The SessionStart hook runs claude_mem_session_start.sh automatically. Manual start only needed if hooks are disabled or worker crashed.

Progressive Disclosure — /mem-search

Always start with the cheapest layer. Only escalate when you need more detail.

LayerCommandCostWhen to Use
1 /mem-search <query> ~50 tokens Always start here. Returns matching observation IDs and one-line summaries.
2 /mem-search timeline <id> ~200 tokens When Layer 1 returns a promising ID. Shows the full session context window.
3 /mem-search fetch <id1> <id2> full content When you need complete content for specific observations. Never fetch all. 
# Layer 1 — find relevant past context
/mem-search auth JWT refresh
  [obs:a1b2c3] PRJ0-42: JWT refresh — httpOnly cookie pattern
  [obs:d4e5f6] PRJ0-55: Token rotation — silent refresh via interceptor

# Layer 2 — explore session context around an observation
/mem-search timeline a1b2c3

# Layer 3 — fetch full content for specific IDs
/mem-search fetch a1b2c3 d4e5f6
Never fetch all

Fetching all memories defeats the purpose and wastes tokens. Always filter first with Layer 1, then fetch selectively.

Citation Format

Reference past observations in prompts using the citation format: [obs:abc123]. This keeps context attribution explicit.

API Endpoints

EndpointMethodPurpose
/mem/search GET Search observations. Query param: q=<query>
/mem/health GET Worker health check. Returns status and version.
/mem/sync POST Manually trigger sync to Brain. Used by claude_mem_stop.sh.

Brain (Postgres)

The Brain is the central, shared memory store. Every agent reads Brain before acting and writes to Brain after completing significant work. Brain data lives in Postgres inside platform/backend/.

Endpoints

EndpointMethodPurpose
/api/v1/brain/memories GET / POST Retrieve or store memories. Scoped to factory or product.
/api/v1/brain/decisions GET / POST Architecture and product decisions with rationale.
/api/v1/brain/patterns GET / POST Reusable patterns proven across products.
/api/v1/brain/conversations GET Conversation history per workflow step (chat/brainstorm/plan/implement).

Read Before Act / Write After Act

  • Before any ticket — run /mem-search <ticket summary> to recall past context
  • Before architecture decisions — query /api/v1/brain/decisions for prior choices
  • After completing a ticket — observations auto-saved by PostToolUse hook
  • After a significant decision — POST to /api/v1/brain/decisions with rationale
  • After session ends — Stop hook promotes score≥0.7 memories to Brain

How They Connect

At session end, the Stop hook runs claude_mem_stop.sh which calls sync_to_brain.py. The bridge evaluates each observation's score and promotes those at or above the threshold to Postgres Brain.

# Manual sync (normally called by Stop hook)
python execution_console/scripts/sync_to_brain.py
  Syncing 47 observations...
  Promoted 12 (score >= 0.7) to Brain
  Skipped 35 (score < 0.7)
Tip

Lower CLAUDE_MEM_BRAIN_THRESHOLD to promote more memories to Brain. Higher threshold keeps Brain focused on high-signal content. Default: 0.7.

Configuration Reference

VariableDefaultPurpose
CLAUDE_MEM_PORT 37777 Port for the claude-mem worker. Override if port conflicts exist.
CLAUDE_MEM_ENABLED true Set to false to disable memory recording (not recommended).
CLAUDE_MEM_DB_PATH ~/.claude-mem/db.sqlite Path to the local SQLite database used by the worker.
CLAUDE_MEM_BRAIN_THRESHOLD 0.7 Minimum score for promoting observations from claude-mem to Postgres Brain.

Hook Configuration

All hooks are configured in .claude/settings.json. See Architecture: Hook System for the full diagram.

EventScriptWhat it does
SessionStart claude_mem_session_start.sh Starts claude-mem worker on CLAUDE_MEM_PORT if not already running.
PostToolUse (Write/Edit) post_tool_use.sh + claude-mem /observe Regenerates graphify-ts knowledge graph, emits execution event, records observation.
PostToolUse (Bash) post_tool_use.sh + claude-mem /observe Emits execution event tagged to CLAUDE_CURRENT_TICKET, records observation.
Stop claude_mem_stop.sh Flushes compression buffer, runs sync_to_brain.py to promote to Brain.

Privacy

Wrap secrets and credentials in <private>...</private> tags in any prompt that may be observed by hooks. The worker strips private content before storage.

# Safe — credentials wrapped in private tags
The API endpoint is /api/v1/products.
<private>API_KEY=sk-prod-xxxx</private>
