Recovery Guide

How ProjectZero handles crashes, failures, and interruptions without losing a single step.

🛡️
Temporal Never Loses State
Every workflow step, every variable, every decision point is persisted in Temporal's event history. Your IDE can crash, your machine can reboot, your network can drop. When you come back, /resume picks up exactly where you left off.

Failure Scenarios

Scenario 1

IDE / Session Crash

Your editor crashes, terminal closes, or machine reboots mid-workflow. The agent session is gone, but the workflow state is safe in Temporal.

Crash detected → /resume reads checkpoint → Continue from last step
Scenario 2

Workflow Failed

An agent step throws an error (test failure, API timeout, invalid output). The retry policy kicks in automatically.

Step fails → Auto-retry (max 3) → Escalate to human
Scenario 3

Context Overflow

The agent's context window fills up during a long implementation. The system checkpoints state, starts a fresh session, and resumes.

Context limit hit → Checkpoint saved → New session → /resume

How Temporal Preserves State

Event Sourcing Under the Hood

Temporal does not store "current state" as a snapshot. Instead, it stores every event that ever happened in the workflow. To reconstruct state, it replays the event history. This makes it impossible to lose data.

1
Activity starts — Temporal records a ActivityTaskScheduled event.
2
Activity completes — Result is stored as ActivityTaskCompleted with the full output payload.
3
Crash happens — The workflow is paused but the event history is intact on the Temporal server.
4
Worker reconnects — Temporal replays all events to rebuild state, then resumes from the exact point of interruption.
5
No duplicate work — Completed activities are not re-executed. Only pending activities resume.

Retry Policy

Each failure class has its own retry behavior configured in the workflow definition:

Failure Class Max Retries Backoff Escalation
Test failure 3 Immediate Block workflow, notify human
API timeout 3 Exponential (1s, 2s, 4s) Mark step as failed, continue next
Agent error (invalid output) 2 10s fixed Escalate to human review
Coverage below threshold 2 Immediate Block at quality gate
Linter failure 3 Immediate Auto-fix attempt, then escalate
Infrastructure (DB, network) 5 Exponential (2s, 4s, 8s, 16s, 32s) Alert ops team
Context overflow 1 N/A Checkpoint + new session

Recovery Commands

/resume
Resume the current workflow from its last checkpoint. Reads the Temporal event history, identifies the last completed step, and continues from there. This is the command you will use most often after any interruption.
/recover-ticket
Recover a specific ticket's workflow by ticket ID. Use when you know the ticket number (e.g., PZ-142) but not the workflow ID. It looks up the associated workflow and resumes it.
/recover-workflow
Recover a workflow by its Temporal workflow ID. Use for direct workflow recovery when you have the workflow ID from the Temporal UI or logs. Supports force-restart of stuck activities.

When to use which

After an IDE crash or context overflow, start with /resume. It auto-detects the active workflow. If you have multiple workflows and need to target a specific one, use /recover-ticket with the ticket ID or /recover-workflow with the workflow ID from the Temporal UI at localhost:8233.
